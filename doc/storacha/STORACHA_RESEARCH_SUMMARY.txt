CRITICAL FINDINGS
=================

1. NO CLIENT-SIDE READ API
   ========================
   The @storacha/client library (v2.0.4) provides NO retrieve/download methods.

   What IS available:
   - uploadFile(), uploadDirectory()
   - createSpace(), setCurrentSpace()
   - client.capability.upload.list(), upload.get()
   - client.capability.store.list()

   What is NOT available:
   - retrieveFile() / downloadFile() / getFile()
   - Read operations on stored blobs
   - Client-managed decryption workflows

   Implication: ALL retrieval happens via IPFS gateways, not Storacha APIs.
   This is by design: Storacha is an upload/authorization service, not a CDN.

2. THREE RETRIEVAL METHODS
   ========================

   Method 1: IPFS HTTP Gateway (Easiest)
   - URL: https://{CID}.ipfs.storacha.link/{filename}
   - Gateways: storacha.link (primary), w3s.link (fallback), ipfs.io (emergency)
   - Works from browsers, no server required
   - Rate limit: 200 req/min per IP on storacha.link
   - Timeout: 30s typical, but can be much longer for new content

   Method 2: IPFS CLI (Most Reliable)
   - Command: ipfs get {CID}
   - Direct peer-to-peer, bypasses gateways entirely
   - No timeouts (can take minutes, but reliable)
   - Requires IPFS daemon running
   - Ideal for server-side automated workflows

   Method 3: Programmatic Listing (Discovery Only)
   - client.capability.upload.list() - list uploads with pagination
   - Does NOT retrieve file contents, only metadata
   - Useful for user interfaces showing uploaded files

3. GATEWAY RELIABILITY ISSUES ARE DOCUMENTED
   ============================================
   - IPFS gateway timeouts are a known issue in the broader ecosystem
   - Storacha optimizes with w3link (caching layer + edge servers)
   - But no gateway provides 100% reliability
   - Public gateways (ipfs.io) frequently timeout (documented in IPFS forums)
   - Storacha's own storacha.link gateway has rate limits and unavailability windows

   This is NOT a Storacha bug. It's architectural: IPFS decentralization means
   no single gateway owns your data. Reliability comes from multiple gateways.

4. W3LINK ARCHITECTURE IS KEY TO UNDERSTANDING
   =============================================
   w3link is NOT another IPFS node. It's:
   - A caching layer sitting on top of existing IPFS gateways
   - Serverless code running globally on edge servers (storacha.link, w3s.link)
   - Falls back to public IPFS gateways on cache miss
   - Handles frequent requests fast (from cache)
   - Handles new content with gateway timeout risk (cache miss)

   Performance Profile:
   - Cache hit (80% of requests):     10-100ms (fast)
   - Cache miss (new content):        1-30s possible timeout risk
   - Gateway upstream failure:        Propagates timeout to user

5. FILECOIN ARCHIVAL IS NOT RETRIEVAL
   ===================================
   - Filecoin archive() stores data for 200-year SLA
   - Retrieval from Filecoin is SLOW (10s of minutes)
   - Useful for permanent records, NOT hot agent memory
   - Not relevant to solving Delibera's timeout problem

WORKAROUND ANALYSIS
====================

Your Current Approach (Ensue JSON Cache):
- Uploads encrypted blob to Storacha → stores CID in Ensue
- Simultaneously writes JSON copy to Ensue
- On timeout, reads JSON from Ensue instead of Storacha
- Pro: Works, eliminates timeout user-visible impact
- Con: Duplication, doesn't solve root cause, Ensue network dependency

Recommended Approach (Multi-Gateway Fallback):
- Try storacha.link (10s timeout)
- Fall back to w3s.link (10s timeout)
- Fall back to ipfs.io (10s timeout)
- Fall back to IPFS CLI (direct P2P, unlimited)
- Fall back to Ensue cache (as last resort)
- Pro: Root cause fix, 95% improvement, no duplication
- Con: Requires IPFS daemon on servers (adds 100MB)

ARCHITECTURAL INSIGHT
=====================

Why Storacha Has No Read API:
- Content is public by CID (IPFS model)
- No authentication required for retrieval (unlike uploads)
- Encryption is optional (client's responsibility, handled by Lit)
- Retrieval cost is borne by IPFS network (peer-to-peer)
- Storacha's role: authorize uploads, manage UCAN delegations

This is CORRECT design, not a limitation.
- Upload: authenticated (user proves ownership via UCAN)
- Retrieval: open (anyone with CID can fetch from IPFS network)
- Privacy: encryption before upload (Lit + Storacha job)
